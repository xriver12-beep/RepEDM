# 電子報管理系統 (Newsletter Management System)

一個功能完整的電子郵件行銷平台，支援電子報的設計、發送、訂閱管理和統計分析。

## 主要功能

- ✅ **電子報設計**: 拖拉式編輯器，支援區塊組裝與 HTML 原始碼編輯。
- ✅ **訂閱者管理**: 完整的會員資料庫、分群功能、批量資料修正與自動退訂管理 (Soft Delete)。
- ✅ **精準受眾計算**: 發送前自動排除無效/退訂/刪除名單，並提供詳細的受眾組成分析 (Total vs Active)。
- ✅ **自動發送**: 排程發送和觸發式郵件。
- ✅ **統計分析**: 開信率、點擊率、退訂率分析，支援視覺化報表與詳細退訂名單紀錄。
- ✅ **追蹤系統**: 內建追蹤像素與連結替換機制，精準追蹤使用者行為。
- ✅ **素材管理**: 集中管理圖片 (JPG/PNG/GIF/SVG) 與文件 (PDF/DOC)，支援分類與使用追蹤。
- ✅ **權限控制**: 多層級使用者權限管理。
- ✅ **流量與速率管制**: 內建 IP 預熱機制、針對 ISP 的發送速率限制 (Throttling) 與網域智慧分流 (Smart Queueing)，確保高送達率並避免被標記為垃圾郵件。

## 角色權限說明 (Role Descriptions)

本系統包含多種使用者角色，其中「經理 (Manager)」與「審核員 (Approver)」的主要差異如下：

### 1. 經理 (Manager)
- **定位**：系統的管理階層，通常為部門主管或團隊負責人。
- **權限**：
  - ✅ **後台存取**：可進入系統後台，查看使用者列表與系統統計數據。
  - ✅ **資料檢視**：可查看下屬及其他使用者的詳細資料。
  - ✅ **簽核職責**：負責審核 **下屬 (直屬員工)** 提交的申請單。
- **適用對象**：部門主管、團隊 Leader。

### 2. 審核員 (Approver)
- **定位**：功能性角色，專注於特定業務流程的簽核。
- **權限**：
  - ❌ **後台存取**：**無法** 進入系統後台，權限等同於一般使用者。
  - ❌ **資料檢視**：僅能查看與自己相關或被指派審核的資料。
  - ✅ **簽核職責**：負責審核 **被指派** 的特定關卡（如財務審核、法務審核），與提交人無層級關係。
- **適用對象**：財務人員、法務人員、總務窗口等特定流程負責人。

| 功能 | 經理 (Manager) | 審核員 (Approver) |
| :--- | :--- | :--- |
| **系統後台** | 可進入 | 不可進入 |
| **簽核邏輯** | 直屬主管簽核 (層級) | 指定關卡簽核 (業務) |
| **資料權限** | 較高 (可視下屬資料) | 一般 (僅限相關資料) |

## 系統功能行為說明 (System Behavior Documentation)

### 1. 訂閱者刪除機制 (Subscriber Deletion)
為了保持資料完整性並避免外鍵約束錯誤 (Foreign Key Constraint)，系統目前採用 **軟刪除 (Soft Delete)** 機制：
- **操作行為**：當使用者執行「刪除」訂閱者時，系統**不會**將資料從資料庫中物理移除。
- **資料狀態**：該訂閱者的狀態欄位 (`status`) 會被更新為 `deleted`。
- **影響範圍**：
  - 「已刪除」的訂閱者將自動從所有未來的行銷活動發送名單中排除。
  - 歷史發送記錄與統計數據將被保留，確保報表準確性。
  - 在訂閱者列表中，預設可能隱藏已刪除的資料（視篩選條件而定）。

### 2. 行銷活動刪除權限 (Campaign Deletion Permissions)
行銷活動的刪除功能受到嚴格的權限控管，規則如下：

- **有權限刪除的角色**：
  - ✅ **管理員 (Admin)**：擁有最高權限，可刪除系統內任何活動。
  - ✅ **經理 (Manager)**：可刪除系統內的活動。
  - ✅ **活動建立者 (Creator)**：無論角色為何（包含一般使用者），只要是該活動的**原始建立者**，皆擁有刪除自己活動的權限。

- **無權限刪除的角色**：
  - ❌ **一般使用者 (User)**：若非該活動的建立者，則無法執行刪除操作。

### 3. 發送流量控制 (Traffic Control)
為確保郵件送達率並維護發送 IP 的信譽 (Reputation)，系統內建了多層次的流量管制機制：

1.  **IP 預熱 (IP Warm-up)**:
    -   針對新 IP 或長期未使用的 IP，系統會自動執行預熱流程。
    -   **策略**: 限制每日發送總量，並以預設 **1.5 倍** 的速率逐日遞增 (例如: 100 -> 150 -> 225...)，直到達到全速發送狀態。
    -   超過當日上限的郵件將自動排程至隔日發送。

2.  **速率限制 (Rate Limiting / Throttling)**:
    -   針對不同 ISP (如 Gmail, Yahoo, Outlook) 設定每分鐘或每小時的接收上限。
    -   **預設範例**: Gmail 每分鐘 100 封，Yahoo 每分鐘 50 封。
    -   避免因短時間內大量連線而被 ISP 暫時封鎖 (421/451 錯誤)。

3.  **智慧分流 (Smart Queueing)**:
    -   發送佇列 (Queue) 採用智慧輪詢機制。
    -   當某個網域 (Domain) 達到速率上限時，系統會自動**暫停**該網域的發送，並**優先處理**其他網域的郵件。
    -   錯開同一網域的發送時間點，實現負載平衡，最大化整體發送效率。

### 4. 取消訂閱機制 (Unsubscribe Mechanism)
系統提供自動化的取消訂閱處理流程，確保遵守相關法規並維護名單品質：

- **雙重檢查機制**: 系統會同時檢查兩個專用信箱 (`noedm@winton.com.tw` 與 `Unsubscribe@winton.com.tw`)。
- **全面掃描**: 自動掃描 `INBOX`、`Junk`、`Spam`、`垃圾郵件` 及自訂資料夾 `EDM_unsubscribe` 中的所有郵件。
- **關鍵字識別**: 支援中英文關鍵字識別 (如 "unsubscribe", "不要再收到", "退訂" 等)，即使用戶使用自然語言回覆也能被正確判讀。
- **自動狀態更新**: 識別成功後，會自動將該訂閱者狀態標記為 `unsubscribed` 並記錄操作時間。

### 5. 發送頻率控管 (Frequency Capping)
系統為防止受眾疲勞 (Audience Fatigue) 並維護發送信譽，提供了自動化的頻率控管機制：

- **運作邏輯**:
  - 系統管理員可設定全域限制，例如「每位訂閱者在 30 天內最多收到 4 封行銷郵件」。
  - 在活動發送前，系統會自動檢查每位受眾過去的發送記錄。
  - 若達到上限，該受眾將被自動標記為 `capped` 並跳過本次發送。

- **Fail-open 機制**:
  - 若頻率檢查服務因故中斷，系統將採取「允許發送」策略，確保重要活動不被阻擋。

- **例外處理 (Force Send)**:
  - 針對重大公告或緊急通知，主管在審核時可勾選「強制發送 (忽略頻率限制)」，確保所有目標受眾皆能收到訊息。

- **可視化回饋**:
  - **活動製作**: 在預覽步驟即時顯示因頻率限制而被排除的人數。
  - **審核詳情**: 主管可查看預計觸及人數與頻率受限人數的對比。
  - **統計報表**: 支援查看因頻率限制而未發送的受眾統計。

## 技術架構

### 前端 (HTML/CSS/JavaScript)
- 靜態 HTML 頁面
- 訂閱者管理介面
- 行銷活動管理
- 統計報表面板
- 審核工作流程

### 後端 (Node.js)
- REST API (Express)
- JWT 認證系統
- 郵件排程服務
- 統計追蹤服務
- 檔案上傳服務

### 資料庫 (MS SQL)
- 訂閱者資料
- 電子報範本
- 發送記錄
- 統計數據

## 專案結構

```
WintonEDM/
├── backend/          # Node.js 後端
├── html-frontend/    # HTML 前端
├── database/         # 資料庫腳本
└── docs/            # 文件
```

## 快速開始

### 環境需求
- Node.js 18+
- MS SQL Server
- Redis (可選，用於快取和佇列)

### 安裝步驟
1. 克隆專案
2. 安裝後端依賴: `cd backend && npm install`
3. 安裝前端依賴: `cd html-frontend && npm install`
4. 配置資料庫連線
5. 啟動服務: `npm run dev`

## 可用頁面 (Available Pages)

啟動服務後，可訪問以下頁面 (預設 HTTP 端口為 **8080**，HTTPS 為 **8443**):

- **儀表板**: http://localhost:8080/dashboard.html
- **訂閱者管理**: http://localhost:8080/subscribers.html
- **行銷活動**: http://localhost:8080/campaigns.html
- **模板管理**: http://localhost:8080/templates.html
- **分析報告**: http://localhost:8080/analytics.html
- **訂閱者統計**: http://localhost:8080/subscriber-stats.html
- **審核工作流程**: http://localhost:8080/approvals.html
- **系統設定**: http://localhost:8080/settings.html
- **取消訂閱頁面**: https://localhost:8443/unsubscribe.html (建議使用 HTTPS)

## 系統架構與端口配置 (System Architecture & Ports)

### 前端伺服器 (Frontend Server)
- **路徑**: `html-frontend/server.js`
- **HTTP 端口**: **8080** (預設)
- **HTTPS 端口**: **8443** (需配置 SSL 憑證)
- **功能**: 提供靜態頁面服務，並作為反向代理將 `/api` 請求轉發至後端。
- **退訂頁面隔離**: `unsubscribe.html` 運行於此伺服器，透過代理與後端通訊，確保安全性。

### 後端伺服器 (Backend Server)
- **路徑**: `backend/src/app.js`
- **HTTP 端口**: **3001** (預設)
- **HTTPS 端口**: **3443** (需配置 SSL 憑證)
- **功能**: 處理業務邏輯、資料庫操作與郵件發送。
- **注意**: 外部請求建議通過前端代理訪問 API，而非直接訪問後端端口。
